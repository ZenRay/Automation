## 一、Pytest 既有问题（非本 Feature 引入）

### 3 个 ERROR：`fixture 'client' not found`

**涉及文件**：
- `tests/test_date_window.py::test_cleanup_execution` (L250)
- `tests/test_date_window_new.py::test_lark_source_date_filter` (L240)
- `tests/test_date_window_new.py::test_cleanup_execution` (L360)

**原因**：这三个测试函数声明了 `client: LarkMultiDimTable` 参数，期望 pytest 注入一个名为 `client` 的 fixture。但项目中没有定义该 fixture（缺少 `conftest.py` 或 fixture 未注册）。这些是需要连接飞书 API 的**集成测试**，不是单元测试。

**修复方案**：
1. 创建 `tests/conftest.py`，定义 `client` fixture：
   ```python
   import pytest
   from automation.client import LarkMultiDimTable

   @pytest.fixture
   def client():
       return LarkMultiDimTable()
   ```
2. 或者将这些测试标记为集成测试，默认跳过：
   ```python
   @pytest.mark.integration
   def test_cleanup_execution(client):
       ...
   ```
   并在 CI 中用 `pytest -m "not integration"` 排除。

### 9 个 WARNING：`PytestReturnNotNoneWarning`

**涉及文件**：
- `tests/test_date_window.py` (多个测试)
- `tests/test_date_window_new.py` (多个测试)
- `tests/test_quality_reject_e2e.py::test_quality_reject_e2e`

**原因**：测试函数使用了 `return True/False` 而非 `assert` 语句。pytest 要求测试函数不返回值，应使用 `assert` 进行断言。

**修复方案**：将 `return True` 改为 `assert <condition>`，将 `return False` 改为 `assert not <condition>` 或 `assert False`。

## 二、GHCR / CI 待完成项

### 1. CI 流水线激活（需推送到仓库）

CI 配置文件已创建，但需要推送到 GitHub 仓库后才能激活：

| 文件 | 触发条件 | 功能 |
|------|---------|------|
| `.github/workflows/ci.yml` | push 到 main/workers/staging/production | lint + test |
| `.github/workflows/deploy.yml` | push 到 staging/production | build + scan + verify + push + smoke |

**操作步骤**：
1. 提交并推送代码到 `workers` 分支
2. 在 GitHub 仓库中创建 `staging` 和 `production` 分支
3. 推送后 `ci.yml` 自动触发 lint + test
4. 合并到 `staging` 后 `deploy.yml` 触发完整构建流水线

### 2. GHCR 推送/拉取验证

**前置条件**：CI 流水线成功推送镜像到 GHCR

**验证步骤**（在部署服务器上执行）：
```bash
# 1. 拉取 CI 构建的镜像
podman pull ghcr.io/<org>/automation:sha-<test-sha>

# 2. 运行验证
podman run --rm \
  -v /opt/automation/config/_lark.ini:/app/automation/conf/_lark.ini:ro \
  -v /opt/automation/config/_maxcomputer.ini:/app/automation/conf/_maxcomputer.ini:ro \
  ghcr.io/<org>/automation:sha-<test-sha> \
  python -c "import automation, workers; print('Podman OK')"

# 3. Compose 启动
IMAGE=ghcr.io/<org>/automation:sha-<test-sha> \
CONFIG_DIR=/opt/automation/config \
podman compose -f deploy/compose.yml -f deploy/compose.staging.yml up -d

podman compose ps
podman compose down
```

### 3. 回滚验证

**前置条件**：GHCR 上至少有 2 个镜像版本

**验证步骤**：
```bash
# 部署当前版本
bash deploy/scripts/deploy.sh deploy --env staging --tag sha-current

# 回滚到上一版本
bash deploy/scripts/deploy.sh rollback --env staging --tag sha-previous

# 确认状态
bash deploy/scripts/deploy.sh status --env staging
# 预期：显示 sha-previous

# 恢复
bash deploy/scripts/deploy.sh deploy --env staging --tag sha-current
```

### 4. Trivy 密钥扫描验证

**位置**：`.github/workflows/deploy.yml` 的 `build` job

trivy 扫描在 CI 环境中自动执行。如需本地验证：

```bash
# macOS: brew install trivy
# Linux: 参考 https://aquasecurity.github.io/trivy/

# 扫描本地镜像
trivy image --scanners secret --exit-code 1 localhost/automation:test
# 预期：exit code 0，无密钥检出
```

### 5. GHCR 包可见性设置

GHCR 包默认继承仓库可见性。如需设置为 public（部署服务器免 token 拉取）：

1. 进入 GitHub 仓库 → Packages → 选择 automation 包
2. Package Settings → Change visibility → Public
3. 设为 public 后，`podman pull` 无需 `GHCR_TOKEN`

如果保持 private，部署服务器需要配置 PAT：
```bash
# 在部署服务器上登录 GHCR
echo $GHCR_PAT | podman login ghcr.io -u <username> --password-stdin
```
